# UI Structure (Sprint 2)

## Цель документа
Описать структуру пользовательского интерфейса приложения Kanban IT:
- экраны системы
- элементы и действия пользователей
- навигацию и переходы

Документ опирается на:
`md_docs/architecture.md`, `md_docs/db_schema.md`, `md_docs/tasks_rules.md`,
`md_docs/wip_rules.md`, `md_docs/workflow.md`.
Новые сущности, статусы, роли или концепции не вводятся.

## Нормативный UI-контракт (технологически-нейтральный)

Данный раздел является источником истины для UI вне зависимости от стека реализации
(Tkinter/Web/другие технологии).

### 1. Канонические сущности и поля представления
- UI оперирует сущностями `pocket`, `project`, `task`, `user`, `status`.
- Для статусов каноническое поле хранения: `status_id`; текстовый `status` допускается как производное представление.
- Все пользовательские действия выполняются над `id` сущностей, а не над display-значениями.

### 2. Контракт навигации и контекста
- Базовый путь контекста: `Карман -> Проект -> Задача`.
- Переход в дочерний контекст обязан сохранять `parent_id` и фильтры текущего контекста.
- Возврат из дочернего контекста обязан восстанавливать последний валидный набор фильтров/выбранных сущностей.

### 3. Контракт UI-состояний
- Для каждого экрана обязательны состояния: `loading`, `empty`, `ready`, `error`.
- Ошибка API отображается пользователю с кодом/текстом ответа без скрытия причины.
- Пустые выборки не считаются ошибкой и отображаются отдельным empty-state.

### 4. Контракт фильтрации и пресетов
- Формат условия: `field + operator + value`.
- Поддерживаемые операторы: `==`, `!=`, `in`, `contains`, `between`, `>`, `<`, `>=`, `<=`.
- Комбинирование условий: линейный список с `AND/OR`, без скобочных групп.
- Пресет по умолчанию: `Активные задачи` (исключает `Завершена`).
- Пресеты персональные и должны сохранять условия, связки и видимость панелей фильтров.

### 5. Контракт RBAC в UI
- Матрица видимости/редактирования в UI обязана соответствовать `architecture.md` и `tasks_rules.md`.
- `executor` работает в read-only для управления статусами задач.
- Операции изменения статуса задач выполняют `curator` и выше.
- Ролевые ограничения UI не могут быть слабее backend RBAC.

### 6. Контракт индикаторов и аналитики
- WIP считается только по задачам в статусе `В работе`.
- Перегрузка отображается как сигнал (индикатор), без автоматического принятия решений системой.
- Приостановки задач отображаются отдельно и учитываются в timeline-семантике.

### 7. Инварианты совместимости
- Допускается platform-specific реализация элементов (кнопки, таблицы, layout), но не изменение семантики действий.
- Любая реализация обязана сохранять контракты ролей, переходов, фильтров, WIP и журнала действий.

### 8. Связь с текущей реализацией
- Tkinter-реализация является текущей референсной реализацией контракта.
- Технические детали Tkinter не являются нормативными, если не влияют на пункты 1-7.

## Иерархия ролей

В системе установлена следующая иерархия ролей (по уровню прав):

admin > head > teamlead > curator > executor

Правило «или выше» означает:
роль, находящаяся левее в иерархии, обладает всеми правами ролей, находящихся правее.

Пример:
- если действие разрешено `curator`, его также могут выполнять `teamlead`, `head`, `admin`;
- если действие разрешено `teamlead`, его также могут выполнять `head`, `admin`.

`executor` не обладает правами вышестоящих ролей.

## Индикатор перегрузки отображается на уровнях:
- задача (подсветка карточки)
- проект (подсветка заголовка проекта)
- карман (подсветка заголовка кармана)

## Журнал действий отображается:
- на уровне задачи (детальная история)
- на уровне проекта и кармана — не отображается в текущей версии UI (данные хранятся системно)

## Экран: Проект (карточка и задачи)

**Назначение:**  
Показать карточку проекта и список задач проекта.

**Роли:**  
`admin`, `head`, `teamlead`, `curator` (полный доступ),  
`executor` (только чтение).

**Элементы:**  
- Панель/Карточка проекта
  - Поля: `id`, `name`, `status`, `date_start`, `date_end`,
    `curator_business_full_name`, `curator_it_full_name`, `pocket_id`
- Таблица задач проекта
  - Поля: `id`, `description`, `status`, `date_created`,
    `date_start_work`, `date_done`, `executor_full_name`, `customer`, `code_link`
- Фильтр по статусу задачи
- Фильтр по исполнителю
- Кнопка/Действие: создать задачу
- Кнопка/Действие: открыть задачу
- Кнопка/Действие: перейти на Kanban-доску (по текущему проекту)
- Кнопка/Действие: перейти на Timeline (по текущему проекту)

**Доступные действия:**  
- создание задачи (`admin`/`head`/`teamlead`/`curator`)  
  - входные данные: описание, заказчик, исполнитель (выбор пользователя), ссылка на код  
  - результат: задача создаётся со статусом `Создана`, сохраняется `executor_user_id`
- фильтрация задач  
  - результат: обновлённый список задач
- переход к задаче  
  - условие: выбран конкретный id  
  - результат: открывается экран «Задача (подробно)»
- переход на Kanban/Timeline  
  - условие: выбран проект  
  - результат: открывается соответствующий экран с фильтром по проекту

**Переходы:**  
- назад к экрану «Проекты кармана (список)»  
- к экрану «Задача (подробно)»  
- к экрану «Kanban-доска»  
- к экрану «Timeline»

## Экран: Задача (подробно)

**Назначение:**  
Показать подробную информацию о задаче и её приостановках.

**Роли:**  
`admin`, `head`, `teamlead`, `curator` (полный доступ),  
`executor` (только чтение).

**Элементы:**  
- Панель/Карточка задачи
  - Поля: `id`, `description`, `status`, `date_created`,
    `date_start_work`, `date_done`, `executor_full_name`, `customer`,
    `code_link`, `project_id`
- Таблица приостановок (task_pauses)
  - Поля: `id`, `date_start`, `date_end`
- Панель/Вкладка: История
  - Список записей журнала по задаче в хронологическом порядке
  - Формат записи: `DD.MM.YYYY HH:MM — ФИО пользователя — Описание действия (Комментарий)`
- Кнопка/Действие: начать работу
- Кнопка/Действие: приостановить
- Кнопка/Действие: возобновить
- Кнопка/Действие: завершить

**Доступные действия:**  

Изменение статуса задачи возможно только в соответствии
с допустимыми переходами из `architecture.md`
и с учётом ролей:

- Начать работу  
  - роли: `curator` и выше  
  - результат: статус → `В работе`

- Приостановить  
  - роли: `curator` и выше  
  - результат: статус → `Приостановлена`

- Возобновить  
  - роли: `curator` и выше  
  - результат: статус → `В работе`

- Завершить  
  - роли: `curator` и выше  
  - результат: статус → `Завершена`

`executor` имеет доступ только на чтение
и не может изменять статус задачи.

Добавление приостановки:
- роли: `curator` и выше  
- входные данные: `date_start`, `date_end`  
- результат: запись в `task_pauses`


**Переходы:**  
- назад к экрану «Проект (карточка и задачи)»

## Дополнение: фильтры задач в Tkinter

### Формат фильтра
- Используется кастомный конструктор условий.
- Каждое условие задаётся как: `поле` + `оператор` + `значение`.
- Поддерживаемые операторы в MVP: `==`, `!=`, `in`, `contains`, `between`, `>`, `<`, `>=`, `<=`.
- Группы со скобками не используются.
- Комбинирование условий: линейный список с выбором связки `AND` или `OR` на строке условия.

### Поведение по умолчанию
- Базовый фильтр при открытии: все задачи, кроме статуса `Завершена`.
- Панель фильтра является скрывающейся (collapse/expand).
- Выбор строки проекта в верхней таблице дэшборда применяет дополнительное условие по `project_id`.

### Пресеты фильтра
- Пресеты включены в MVP.
- Пресеты являются персональными (для текущего пользователя).
- Доступные действия: `Сохранить текущий`, `Применить`, `Переименовать`, `Удалить`.
- Сохраняются: список условий, связки `AND/OR`, сортировка, состояние панели (свернута/развернута).
- Предустановленный пресет по умолчанию: `Активные задачи`.

## Нормативное уточнение: Tkinter UI (замена web UI)

### Навигация и компоновка главного окна
- Верхняя панель навигации горизонтальная, в стиле оконных приложений ОС.
- Кнопки рабочих зон размещаются сразу под верхним меню и работают как лента:
  - `Дэшборд`
  - `Kanban`
  - `Timeline`
  - `Аналитика`
- Web UI не является целевым интерфейсом для этой версии.

### Дэшборд: структура и взаимодействие
- Дэшборд содержит две таблицы.
- Верхняя таблица: объединённые данные кармана и проекта.
- Нижняя таблица: задачи.
- При выделении строки проекта в верхней таблице нижняя таблица фильтруется по этому проекту.
- Открытие карточки проекта выполняется отдельной кнопкой в строке верхней таблицы.

#### Верхняя таблица (карман + проект): поля
- `pocket_id`
- `pocket_name`
- `project_id`
- `project_name`
- `project_code` (текст)
- `date_start`
- `date_end`
- `status`
- `owner`
- `owner_it`

#### Нижняя таблица (задачи): поля
- `id`
- `description`
- `status`
- `date_created`
- `date_start_work`
- `date_done`
- `executor_full_name`
- `customer`
- `code_link`

### Фильтрация задач на дэшборде
- По умолчанию показываются все задачи, кроме статуса `Завершена`.
- Панель фильтров скрывающаяся.
- Используется кастомный конструктор фильтров: `поле` + `оператор` + `значение`.
- Поддерживаемые операторы: `==`, `!=`, `in`, `contains`, `between`, `>`, `<`, `>=`, `<=`.
- Группы со скобками не используются.
- Условия формируются линейным списком со связками `AND/OR`.

### Пресеты фильтров (MVP)
- Пресеты включены в MVP.
- Пресеты персональные для текущего пользователя.
- Действия: `Сохранить текущий`, `Применить`, `Переименовать`, `Удалить`.
- Сохраняются: условия фильтра, связки `AND/OR`, сортировка, состояние панели фильтров.
- Пресет по умолчанию: `Активные задачи`.

### Создание и редактирование сущностей
- Создание новых сущностей и правка справочников выполняются в pop-up окнах.
- Для редактирования записи в таблицах используется кнопка `Изменить` в строке.
