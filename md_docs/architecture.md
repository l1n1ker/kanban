# Архитектура системы (Sprint 2)

## 1. Цель документа

Документ фиксирует целевую архитектуру системы Kanban IT для Sprint 2.
Документ является нормативным источником истины для реализации.

## 2. Базовые принципы

- Система фиксирует факты и визуализирует состояние.
- Система не принимает управленческие решения за человека.
- Приоритеты и рекомендации системой не формируются.
- Ролевая модель и ограничения действий обязательны.

## 3. Архитектура исполнения

Целевая архитектура:

- интерфейс: desktop-приложение на Tkinter;
- backend: локальный FastAPI-процесс на рабочей станции пользователя;
- хранилище: SQLite-файл на сетевом ресурсе;
- путь к БД задается в локальных настройках клиента.

Компоненты:

```
[ Пользователь (Windows) ]
            |
            v
[ Tkinter UI ]
            |
            v
[ Local FastAPI Backend ]
            |
            v
[ SQLite (network share) ]
```

Web UI допускается как legacy-компонент, но не является целевым интерфейсом Sprint 2.

## 4. Аутентификация и роли

### 4.1 Аутентификация

- Используется логин текущей Windows-сессии.
- Пароли в приложении не используются.
- Сопоставление пользователя выполняется по `users.login`.

### 4.2 Роли

- `admin`
- `head`
- `teamlead`
- `curator`
- `executor`

Иерархия прав:

`admin > head > teamlead > curator > executor`

## 5. Модель предметной области

Основные сущности:

- пользователь (`users`)
- карман (`pockets`)
- проект (`projects`)
- задача (`tasks`)
- приостановка задачи (`task_pauses`)
- журнал действий (`action_log`)
- единый справочник статусов (`statuses`)

Статусы всех сущностей хранятся через `status_id` (FK на `statuses`).

## 6. Правила процессов

### 6.1 Карман

- Логическая группа проектов.
- Статусы кармана задаются справочником `statuses` с `entity_type = pocket`.

### 6.2 Проект

- Создается внутри кармана.
- Статусы проекта задаются справочником `statuses` с `entity_type = project`.
- Завершение проекта доступно роли `curator` и выше.

### 6.3 Задача

- Создается только внутри проекта.
- На старте у задачи один исполнитель.
- Статусы задачи задаются справочником `statuses` с `entity_type = task`.
- Допустимые переходы:
  - `Создана -> В работе`
  - `В работе -> Приостановлена`
  - `Приостановлена -> В работе`
  - `В работе -> Завершена`
- Запрещенные переходы:
  - `Создана -> Завершена`
  - `Приостановлена -> Завершена`

## 7. WIP и перегрузка

- Активный WIP: задачи в статусе `В работе`.
- Перегрузка допускается, блокировки отсутствуют.
- Система только подсвечивает перегрузку на уровнях задачи, проекта и кармана.

## 8. Учет времени

- Учет в календарных днях.
- Приостановки фиксируются датой начала и окончания.
- Приостановки вычитаются из фактического срока выполнения задачи.

## 9. Журнал действий

Логирование обязательно и append-only.

`action_type`:

- `create`
- `update_status`
- `assign`
- `pause_start`
- `pause_end`
- `close`
- `edit`

Лог не влияет на бизнес-логику и WIP.

## 10. Ограничения

Система не включает:

- автоматическую приоритизацию;
- автоматические рекомендации;
- автоматическое управление планом;
- прогнозирование и оптимизацию.
