# Архитектура системы (Sprint 1)

## Роли системы

- admin  
  Техническая роль.
  Имеет полный доступ ко всем экранам и справочникам.


- head  
  Руководитель подразделения.
  Просмотр всех карманов, проектов, задач, журналов.
  Может вмешиваться в управление при необходимости.

- teamlead  
  Управляет нагрузкой исполнителей.
  Имеет право изменять задачи и статусы в пределах доступных проектов.

- curator  
  Управляет проектами и задачами своего проекта.

- executor  
  Исполняет задачи.
  Не управляет чужими задачами.

## Иерархия ролей (от большего каоличества прав к меньшему)

admin > head > teamlead > curator > executor

## Правила логирования

Каждое логируемое действие должно соответствовать одному `action_type`.

Соответствие:

- создание сущности → create
- изменение статуса → update_status
- назначение / смена ответственного → assign
- начало приостановки → pause_start
- завершение приостановки → pause_end
- завершение сущности → close
- редактирование атрибутов → edit

old_value / new_value:
- хранят краткое человекочитаемое описание
- формат свободный, текстовый
- предназначены для чтения человеком, не для обработки системой

## Создание и завершение карманов и проектов

### Карман
- создаётся пользователем с ролью `admin` или `head`
- обязательные поля: наименование, даты, подразделение, ответственный
- статус при создании: `Запущен`
- завершение — ручное действие, статус `Завершён`

### Проект
- создаётся в рамках существующего кармана
- создаётся `curator` или выше
- обязательные поля: наименование, карман, кураторы, даты
- статус при создании: `Активен`
- завершение — ручное действие, статус `Завершён`
- Проект может завершать: curator и выше


## 1. Цель документа

Данный документ фиксирует архитектурные решения Sprint 1 **без реализации кода**.
Архитектура должна:

* поддерживать принятую процессную модель
* не навязывать управленческие решения
* быть минимальной и расширяемой

Документ является нормативным и используется Codex как источник истины.

---

## 2. Общая схема

Система представляет собой:

* локальное web-приложение (Streamlit)
* с доступом по Windows-аутентификации
* с общей БД SQLite, размещённой на сетевом диске

Компоненты:

```
[ Пользователь (Windows) ]
            |
            v
[ Streamlit UI ]
            |
            v
[ Application Logic ]
            |
            v
[ SQLite (network share) ]
```

---

## 3. Аутентификация и роли

### 3.1. Аутентификация

* Используется логин текущей Windows-сессии
* Пароли не используются
* Логин читается из окружения ОС
* Логин хранится в справочнике пользователей и сопоставляется по `users.login`

### 3.2. Роли (логические)

Роли определяются **данными**, а не механизмами безопасности:

* Администратор (admin)
* Начальник отдела (head)
* Тимлид (teamlead)
* Куратор проекта (curator)
* Исполнитель (executor)

Роль влияет на:

* доступные действия
* доступные экраны

## 4. Сущности и минимальные атрибуты

### 4.0. Пользователь

Минимальные поля:

* id
* login
* full_name
* role (admin / head / teamlead / curator / executor)

### 4.1. Карман

Минимальные поля:

* наименование
* дата_начала
* дата_окончания

* дата_окончания

### 4.3. Задача

Минимальные поля:

* id
* описание
* id_проекта
* статус
* дата_создания
* дата_начала_работы
* дата_завершения

---

## 5. Статусы и переходы задач

### 5.1. Статусы

* Создана
* В работе
* Приостановлена
* Завершена

### 5.2. Допустимые переходы

```
Создана -> В работе
В работе -> Приостановлена
Приостановлена -> В работе
В работе -> Завершена
```

Запрещены:

* Создана -> Завершена
* Приостановлена -> Завершена

---

## 6. Учёт времени

* Все даты учитываются в календарных днях
* Время (часы/минуты) не используется
* Приостановки имеют дату начала и окончания
* Фактический срок = (дата_завершения - дата_начала_работы) минус дни приостановки
* Запрет на использование времени (часы/минуты) относится к задачам.
* Системное логирование (timestamp в action_log) разрешено и не противоречит данной политике.
* Запрет на часы/минуты относится только к задачам.
* Системное логирование (timestamp) разрешено.

---

## 7. WIP и перегрузка

* WIP считается по количеству задач со статусом "В работе"
* У задачи всегда один исполнитель
* Перегрузка:

  * допускается
  * визуализируется
  * не блокируется

Система не предлагает решений, только отображает состояние.

## Журнал действий

Логирование является обязательной частью архитектуры.
Журнал действий обязателен для всех сущностей.
Отображение журнала в UI является опциональным
и определяется документом ui_structure.md.

Назначение:
* восстановление хронологии изменений
* анализ перегрузок и конфликтов
* объяснение управленческих решений

Не используется для:
* контроля активности пользователей
* автоматического управления процессами
* расчётов, оптимизаций и рекомендаций

Принципы:
* логируются только изменения состояния системы
* чтение данных и навигация не логируются
* лог append-only, неизменяемый и не удаляемый
* лог не влияет на бизнес-логику и WIP
* записи читаемы человеком, без технических структур
