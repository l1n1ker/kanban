## Принцип совместимой нормативности

Для изменений в проекте действует правило:
- `md_docs` определяет целевое нормативное состояние.
- Рабочее поведение системы не ломается без отдельного явного решения.
- При конфликте между нормой и работающим сценарием используется временный `Compat bridge`.
- Для каждого `Compat bridge` фиксируются: причина, риски, срок снятия, владелец.
- Изменения принимаются только при одновременном выполнении:
  - regression-тестов текущего поведения;
  - compliance-тестов целевой нормы.
- После прохождения переходного периода `Compat bridge` удаляется, норма становится строгой.

Текущий статус bridge `status -> status_id` (зафиксировано 2026-02-21):
- в хранении bridge закрыт: канон только `status_id`;
- в HTTP API поле `status` временно оставлено как deprecated compatibility field на 1 релиз (derived from `status_id`).

## Состав

### 1. tasks_rules.md
Правила создания, изменения, приостановки и завершения задач.
Определяет жизненный цикл задачи и ответственность ролей.

### 2. wip_rules.md
Правила учёта нагрузки (WIP).
Описывает, как система выявляет перегрузку,
но **не управляет приоритетами автоматически**.

### 3. workflow.md
Сквозной процесс работы:
Карман → Проект → Задача.
Фиксирует границы ответственности человека и системы.

## Ключевые принципы
- Все проекты стартуют всегда
- Приоритеты отсутствуют намеренно
- Система не принимает управленческих решений
- Основная цель — прозрачность и выявление перегрузки

Любые изменения логики сначала отражаются в документации,
и только затем — в коде.

## UI (Tkinter + локальный FastAPI)
- Основной интерфейс: desktop-клиент на Tkinter.
- Backend запускается локально на рабочей станции пользователя.
- Локальный backend работает с SQLite-файлом по пути из настроек клиента (сетевой путь).
- Web UI (`/ui`) рассматривается как legacy и не является целевым интерфейсом.
